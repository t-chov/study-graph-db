# 学習メモ

## Mutex について

- Mutex をとっている. これは同時アクセスの安全性のため. Go の map は並行安全性がないので、別の goroutine による操作を抑制しなければ壊れる
- 実プロダクトにおいても、並行安全性は課題となりうる
    - 整合性: ノード/エッジ本体更新とインデックス更新がズレると、検索結果が壊れる
    - 原子性: 「データ更新 + 全インデックス更新」を1トランザクションとして扱う必要がある
    - 隔離性: 同時トランザクション間で、どの時点のインデックスを見せるか（MVCC/ロック戦略）
    - 回復性: クラッシュ時に、WALやリカバリでデータとインデックスを再同期する必要がある
    - 性能トレードオフ: 強い一貫性を取るほどロック競合が増えやすい

## インデックス構築について

- Naive な実装ではインデックス構築が O(NM) のような計算量になるが、これは工夫で回避できる

## 2-hop クエリ高速化の観点

- 例: `User(alice)` をフォローしているユーザーが `SUBSCRIBES` しているプロジェクトを知りたい
- この種のクエリは、全エッジ走査ではなく「始点から局所的にたどる」構造が重要
- 必要なデータ構造の中心は typed adjacency:
  - `inAdjByType[nodeID][edgeType] -> neighborNodeIDs`
  - `outAdjByType[nodeID][edgeType] -> neighborNodeIDs`
- さらにノード特定のために、`User.name=alice` のようなプロパティ索引（v2）を入口に使う
- 実行イメージ:
  1. `alice` を索引で `aliceID` に解決
  2. `inAdjByType[aliceID]["FOLLOWS"]` でフォロワー集合を取得
  3. 各フォロワーの `outAdjByType[follower]["SUBSCRIBES"]` をたどってプロジェクトを集約
- これにより、計算量は「グラフ全体サイズ」ではなく「実際にたどる近傍サイズ」に寄せられる

## データ構造と探索性能の理解整理

- Production の Graph DB では隣接リスト（index-free adjacency に近い構造）を中心に、インデックス、ID管理、ページ管理、Tx/WAL などを組み合わせる
- ノード削除も「向いているエッジを消す」だけでは不十分で、インデックス更新・制約・トランザクション整合性・回復性まで考慮が必要
- 探索性能は BFS/DFS の枝刈りだけでなく、次の総合で決まる:
  - 開始点を絞るインデックス
  - typed adjacency などのデータ構造
  - クエリプランナーの探索順選択
  - 実行時のキャッシュや I/O 特性

## トランザクション実装方式の整理

- 学習実装の v8 は「全体スナップショット方式」:
  1. Begin 時に現在状態をコピー
  2. Tx 中の変更はコピー側にのみ適用
  3. Commit で本体へ反映、Rollback でコピー破棄
- この方式は理解しやすい反面、メモリコストが大きい（大きなグラフではほぼ 2 倍以上になりうる）
- 実運用DBでは全体コピーは避け、差分管理 + 細粒度ロック/MVCC を使うことが多い
- Graph DB のロック粒度は「行」に限らず、ノード/エッジ/インデックス項目/ページ単位などになる
- さらに WAL を組み合わせて、クラッシュ時の回復性と durability を担保する
